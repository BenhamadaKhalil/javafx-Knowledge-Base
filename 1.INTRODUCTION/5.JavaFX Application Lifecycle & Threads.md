# 🎬 Lesson 3 – JavaFX Application Lifecycle & Threads

## 🧠 Overview

In this lesson, we’ll explore how the **JavaFX Runtime** manages your application behind the scenes — including the **threads** it creates and the **lifecycle methods** it calls during execution.

JavaFX applications don’t just run in one simple thread. The JavaFX runtime launches multiple threads that perform different roles to ensure smooth UI behavior and responsiveness.

---

## ⚙️ JavaFX Threads Overview

When a JavaFX application starts, the **`launch()`** method (called from your `main()` method) creates **two important threads**:

|Thread|Purpose|
|---|---|
|🟢 **Launcher Thread**|Starts the JavaFX application and calls `init()`.|
|🔵 **Application Thread (JavaFX Application Thread)**|Runs the UI — creates the stage, scene, and handles user interaction.|

> ⚠️ The **JavaFX Application Thread** is the **main UI thread**.  
> Anything that interacts with UI elements must run here.

---

## 🧩 The JavaFX Application Lifecycle

The JavaFX runtime calls **four main methods** in sequence during the application lifecycle:

|Stage|Method|Description|Thread|
|---|---|---|---|
|1️⃣|**Constructor** (`HelloApplication()`)|Automatically created when the app starts. Used for simple setup.|Application Thread|
|2️⃣|**`init()`**|Used for pre-initialization logic (e.g., connecting to databases, loading resources).|Launcher Thread|
|3️⃣|**`start(Stage stage)`**|The main entry point for your app’s UI (create stage, scene, buttons, etc.).|Application Thread|
|4️⃣|**`stop()`**|Called when the app closes — cleanup, save data, etc.|Application Thread|

---

## 💡 Exploring the Application Class

The `Application` class defines all these methods — but they’re **empty** by default.  
That’s why we **override** them in our own class.

To see this yourself:

1. Hover over `Application`.
    
2. Click _"Go to Source"_ or _"Decompile Class"_ in IntelliJ.
    
3. You’ll see:
    
    ```java
    public class Application {
        public Application() {}
        public void init() throws Exception {}
        public abstract void start(Stage stage) throws Exception;
        public void stop() throws Exception {}
    }
    ```
    

---

## 🔍 Understanding the `init()` Method

- The `init()` method is called **before** the `start()` method.
    
- It’s executed on the **Launcher Thread**, not the Application Thread.
    
- It’s used for:
    
    - Preloading data
        
    - Connecting to services
        
    - Any setup tasks before showing the UI
        

### ⚠️ Important Rules

- ❌ You **cannot** create a `Stage` or `Scene` inside `init()`.
    
- ✅ You **can** create **non-visual** JavaFX objects like `Button`, `Label`, etc.
    
- 💡 Think of it as the **setup phase** before the app window appears.
    

---

## 🧪 Example: Overriding the `init()` Method

```java
@Override
public void init() throws Exception {
    System.out.println("Before the app starts...");
    Thread.sleep(5000); // Simulate a long initialization task
}
```

When you run this:

- The message appears immediately.
    
- The app **waits 5 seconds** before showing the window.
    
- This simulates a process like connecting to a database.
    

---

## 🧵 Checking Which Thread Runs What

You can verify the current thread using:

```java
System.out.println(Thread.currentThread().getName());
```

Add this in both `init()` and `start()`:

```java
@Override
public void init() throws Exception {
    System.out.println("init() → " + Thread.currentThread().getName());
}

@Override
public void start(Stage stage) {
    System.out.println("start() → " + Thread.currentThread().getName());
}
```

🧩 Output example:

```
init() → JavaFX-Launcher
start() → JavaFX Application Thread
```

---

## ⚠️ Why Separate Threads?

If your app runs **heavy tasks** (e.g., downloading data, connecting to a remote server) on the **UI thread**, it can **freeze** the interface — leading to the classic “Not Responding” window.

### Example Problem: Blocking the UI

```java
Button blockButton = new Button("Block");
blockButton.setOnAction(e -> {
    System.out.println("UI now blocked...");
    try {
        Thread.sleep(10000); // 10 seconds of blocking
    } catch (InterruptedException ex) {
        ex.printStackTrace();
    }
});
```

When you click this:

- The UI freezes for 10 seconds.
    
- You can’t click any other buttons.
    
- After 10 seconds, the UI updates again.
    

This happens because both the **sleeping task** and **UI updates** share the same **Application Thread**.

---

## ✅ Solution: Use Threads

To prevent blocking:

- Run long-running tasks (like DB connections or network calls) in a **separate thread**.
    
- Keep your UI responsive.
    
- Once the background task finishes, use **Platform.runLater()** to update the UI safely.
    

You’ll learn this in detail in the next lesson.

---

## 🧠 Summary

|Concept|Description|
|---|---|
|`launch()`|Starts the app, creating both launcher and application threads.|
|`init()`|Called on the Launcher Thread — setup logic (no UI).|
|`start(Stage)`|Runs on the Application Thread — main UI logic.|
|`stop()`|Called before the app closes — cleanup logic.|
|Separate Threads|Keep the UI responsive during long tasks.|

---

## 💬 Key Takeaways

- JavaFX automatically manages multiple threads.
    
- `init()` and `start()` run on **different threads**.
    
- Don’t do heavy work in `start()`.
    
- Learn to use **background threads** for heavy tasks.
    
- Always update UI components from the **Application Thread**.
